Pyreon's reactivity system is the foundation of the framework. It provides fine-grained primitives
that track dependencies automatically — no dependency arrays, no manual subscriptions.

## How Tracking Works

When you call a signal inside a tracking context (an `effect`, `computed`, or reactive JSX
expression), Pyreon records that dependency. When the signal's value changes, only the subscribers
that read it are re-run.

<Steps>
<Step>

**Create a signal** — a reactive container for a value.

```ts
const count = signal(0)
```

</Step>
<Step>

**Read it inside a tracking context** — the context subscribes to `count`.

```ts
effect(() => {
  console.log("count is", count())  // runs now, and again each time count changes
})
```

</Step>
<Step>

**Write to it** — all subscribers re-run automatically.

```ts
count.set(1)   // logs "count is 1"
count.set(2)   // logs "count is 2"
```

</Step>
</Steps>

<Callout type="info" title="No dependency arrays">
  Pyreon tracks dependencies at runtime by intercepting signal reads. You never write
  `[count, name]` — if you read a signal, you depend on it. If you don't, you don't.
</Callout>

## Core Primitives

<Tabs items={["signal", "computed", "effect"]}>
<Tab value="signal">

### `signal<T>(initialValue)`

A writable reactive value. Reading returns the current value. Writing notifies subscribers.

```ts title="src/counter.ts"
import { signal } from "@pyreon/reactivity"

const count = signal(0)

count()           // read → 0
count.set(1)      // write → notifies subscribers
count.update(n => n + 1)  // update with a function
```

<Tab value="computed">

### `computed<T>(fn)`

A derived reactive value. Recomputes when any signal it reads changes. Cached between reads.

```ts title="src/derived.ts"
import { signal, computed } from "@pyreon/reactivity"

const count = signal(2)
const doubled = computed(() => count() * 2)

doubled()   // 4
count.set(5)
doubled()   // 10 — recomputed lazily
```

<TypeTable
  type={{
    "computed(fn)": {
      description: "Creates a computed value derived from other signals.",
      type: "ReadonlySignal<T>",
    },
    "computed()": {
      description: "Reads the current derived value. Recomputes only if a dependency changed.",
      type: "T",
    },
  }}
/>

<Callout type="info" title="Lazy evaluation">
  Computed values are lazy — they only recompute when read, not when a dependency changes.
  This makes chains of computed values efficient.
</Callout>

</Tab>
<Tab value="effect">

### `effect(fn)`

Runs a side effect when dependencies change. Returns a disposer.

```ts title="src/sync.ts"
import { signal, effect } from "@pyreon/reactivity"

const name = signal("Alice")

const dispose = effect(() => {
  document.title = `Hello, ${name()}`  // runs immediately and on every change
})

name.set("Bob")   // document.title → "Hello, Bob"

dispose()         // stops the effect
```

/>

</Tab>

## Additional Utilities

### `batch(fn)`

Groups multiple signal writes into a single flush. Subscribers run once after all writes.

```ts title="src/batch-example.ts"
import { signal, effect, batch } from "@pyreon/reactivity"

const x = signal(1)
const y = signal(2)

effect(() => console.log(x() + y()))  // logs 3

batch(() => {
  x.set(10)
  y.set(20)
})
// logs 30 once — not 12, then 30
```

<Callout type="info">
  Event handlers in Pyreon's JSX renderer run inside a batch automatically. You only need
  explicit `batch()` calls in your own async code.
</Callout>

---

### `createSelector<T>(source)`

Optimizes a pattern where many items each need to know if they are "selected". Without
`createSelector`, every item re-renders on every selection change. With it, only the
previously selected and newly selected items update.

```ts title="src/list.ts"
import { signal, createSelector } from "@pyreon/reactivity"

const selectedId = signal<string | null>(null)
const isSelected = createSelector(selectedId)

// Inside a For loop — only 2 items re-evaluate per click
<For
  each={items}
  key={i => i.id}
  children={item => (
    <li class={() => isSelected(item.id) ? "selected" : ""}>
      {item.name}
    </li>
  )}
/>
```

---

### `runUntracked(fn)`

Reads signals without creating a dependency.

```ts
import { runUntracked } from "@pyreon/reactivity"

effect(() => {
  const current = runUntracked(() => count())  // read without subscribing
  console.log("triggered, but count is", current)
})
```

## Gotchas

<Callout type="warn" title="Don't destructure signals">
```ts
const { x, y } = position  // position is a signal — this reads x and y ONCE
// x and y are now plain values, not reactive

// Correct: read inside a tracking context
const x = () => position().x
```
</Callout>

<Callout type="warn" title="Async functions break tracking">
```ts
effect(async () => {
  const a = count()  // ✓ tracked — synchronous read
  await fetch("/api")
  const b = count()  // ✗ NOT tracked — after the await, tracking context is gone
})
```

Use `runUntracked` or restructure to read all signals synchronously before any `await`.
</Callout>

<Callout type="warn" title="Object/array mutations are not reactive">
```ts
const items = signal([1, 2, 3])
items().push(4)   // ✗ mutates the array — subscribers don't see the change
items.set([...items(), 4])  // ✓ set a new array reference
```
</Callout>
