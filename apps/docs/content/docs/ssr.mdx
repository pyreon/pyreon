---
title: "Server-Side Rendering"
description: "renderToString, renderToStream, and hydrateRoot for server-side rendering."
---

`@nova/runtime-server` provides `renderToString` and `renderToStream` for server-side rendering.
The client uses `hydrateRoot` from `@nova/runtime-dom` to attach event listeners without
re-building the DOM.

## Installation

```bash title="Terminal"
bun add @nova/runtime-server @nova/runtime-dom
```

## Full SSR Setup

<Steps>
<Step>

### Server: render to HTML

<Tabs items={["renderToString", "renderToStream"]}>
<Tab value="renderToString">
```ts title="server.ts"
import { renderToString } from "@nova/runtime-server"
import App from "./src/App"

const html = await renderToString(<App />)

// Send as HTTP response
const response = new Response(
  `<!doctype html>
   <html>
     <head><title>Nova SSR</title></head>
     <body>
       <div id="app">${html}</div>
       <script type="module" src="/src/main.tsx"></script>
     </body>
   </html>`,
  { headers: { "Content-Type": "text/html" } }
)
```
</Tab>
<Tab value="renderToStream">
```ts title="server.ts"
import { renderToStream } from "@nova/runtime-server"
import App from "./src/App"

// Returns a ReadableStream — ideal for streaming SSR
const stream = renderToStream(<App />)

return new Response(stream, {
  headers: { "Content-Type": "text/html" },
})
```
</Tab>
</Tabs>

</Step>
<Step>

### Client: hydrate instead of mount

```ts title="src/main.tsx"
import { hydrateRoot } from "@nova/runtime-dom"
import App from "./App"

hydrateRoot(document.getElementById("app")!, <App />)
```

<Callout type="info">
  `hydrateRoot` walks the existing DOM in parallel with the VNode tree. It reuses
  existing elements, only attaching event listeners and reactive effects — no DOM is
  rebuilt.
</Callout>

</Step>

## How Hydration Works

Nova uses a **walk-and-claim** strategy — no special SSR markers are needed in the HTML.

| Content | Strategy |
|---|---|
| Static elements | Match by tag position, attach event listeners + reactive prop effects |
| Reactive text `() => string` | Find existing text node, attach reactive effect to `.data` |
| Components | Run component fn, recurse into returned VNode + DOM child |
| Reactive blocks `() => VNode` | Insert comment marker, attach `mountReactive` |
| `<For>` lists | Full remount — can't map keys to DOM without markers |
| `<Portal>` | Always remounts into target |

<Callout type="warn" title="For lists always remount during hydration">
  `<For>` cannot safely match keyed items to server-rendered DOM without SSR markers.
  List content is remounted on the client. This is transparent to the user but means
  the DOM flickers briefly for long lists.
</Callout>

## Data Fetching with SSR

For data that needs to be available on both server and client, fetch during SSR and
serialize into the HTML:

```ts title="server.ts"
const data = await fetch("https://api.example.com/items").then(r => r.json())
const html = await renderToString(<App items={data} />)

const fullHtml = `
  <!doctype html>
  <html>
    <body>
      <div id="app">${html}</div>
      <script>window.__SSR_DATA__ = ${JSON.stringify(data)}</script>
      <script type="module" src="/src/main.tsx"></script>
    </body>
  </html>
`
```

```ts title="src/main.tsx"
const ssrData = (window as { __SSR_DATA__?: unknown }).__SSR_DATA__
hydrateRoot(document.getElementById("app")!, <App items={ssrData} />)
```
