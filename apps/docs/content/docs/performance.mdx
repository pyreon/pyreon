Pyreon is designed from the ground up for minimal DOM work. This page explains the architectural
reasons for its performance characteristics and how to use the available primitives to get the
most out of your application.

<Callout type="info" title="You usually don't need to optimize">
  Signal-based reactivity is already highly granular. Before reaching for advanced APIs,
  profile with browser DevTools to confirm there is actually a bottleneck.
</Callout>

## How Pyreon Works

Unlike React and Vue which use a virtual DOM, Pyreon updates the real DOM directly through
signal subscriptions. Here is the full pipeline from state change to screen update:

### 1. Components run once

When a component mounts, Pyreon calls it exactly once. The JSX return value creates real DOM
nodes — not a virtual tree to be diffed later. All subsequent updates bypass the component
function entirely.

### 2. Signals notify subscribers directly

When a signal changes, it notifies only the specific DOM bindings that read it. There is no
tree walk, no diffing, no reconciliation. A text node bound to `name()` updates when `name`
changes — nothing else runs.

### 3. Compiler-driven template emission

The Pyreon compiler detects static JSX element trees and emits optimized `_tpl()` calls
instead of nested `h()` calls. This unlocks three key optimizations:

| Optimization | What it does | Savings |
|---|---|---|
| `cloneNode(true)` | Browser clones a pre-parsed `<template>` element in one C++ call | Replaces N `createElement` + `setAttribute` calls |
| `TextNode.data` | Updates existing text nodes in-place via `.data` property | Avoids `.textContent` which destroys and recreates child nodes |
| `_bind()` (static-dep tracking) | Tracks signal dependencies once at creation, never re-tracks | Eliminates per-update cleanup, re-registration, and tracking context overhead |

### 4. Zero-allocation update path

On a signal change, the update path is:

```
signal.set(newValue)
  → notifySubscribers (iterate Set, call each fn)
    → _bind's run() — just calls fn(), no bookkeeping
      → fn() reads signal (activeEffect is null → trackSubscriber is a no-op)
      → fn() writes to DOM node (textNode.data = value)
```

No objects allocated. No arrays created. No maps consulted. Just function calls down to a DOM write.

## Architectural Comparison

| Technique | Pyreon | React | Vue | SolidJS |
|---|---|---|---|---|
| Rendering model | Direct DOM via signals | Virtual DOM diffing | Virtual DOM + patch flags | Direct DOM via signals |
| Component execution | Once at mount | Every state change | Once (setup), but re-renders template | Once at mount |
| Update granularity | Single DOM node | Component subtree | Component subtree (optimized) | Single DOM node |
| Template strategy | `cloneNode` from `<template>` | `createElement` per element | `cloneNode` (vapor mode) | `cloneNode` from `<template>` |
| Dependency tracking | Static (track once via `_bind`) | Manual deps arrays | Proxy-based (automatic) | Dynamic (re-track each run) |
| List diffing | LIS algorithm, O(n log k) | O(n) reconciliation | O(n) with patch flags | LIS algorithm |
| Selection optimization | `createSelector` — O(1) | None built-in | None built-in | `createSelector` — O(1) |

execution order). All frameworks use equivalent optimizations: template cloning for Pyreon
and Solid, `memo` for React, `createSelector` where available.

|---|---|---|---|---|---|---|
| Create 1,000 rows | 8 ms | 9 ms | 10 ms | 11 ms | 14 ms | 33 ms |
| Replace 1,000 rows | 8 ms | 10 ms | 10 ms | 11 ms | 14 ms | 31 ms |
| Partial update (every 10th) | 1 ms | 5 ms | 5 ms | 7 ms | 9 ms | 6 ms |
| Select row | 0 ms | 5 ms | 5 ms | 5 ms | 8 ms | 8 ms |
| Swap rows | 1 ms | 5 ms | 5 ms | 5 ms | 8 ms | 18 ms |
| Clear rows | 0 ms | 5 ms | 5 ms | 5 ms | 8 ms | 8 ms |
| Create 10,000 rows | 79 ms | 103 ms | 104 ms | 131 ms | 289 ms | 540 ms |

### Slowdown vs Best Framework

Lower is better. 1.00x means fastest among frameworks (excluding vanilla JS baseline).

| Operation | Pyreon | SolidJS | Vue 3 | Preact | React 19 |

| Replace 1,000 rows | **1.00x** | 1.03x | 1.21x | 1.49x | 3.14x |

<Callout type="info" title="Methodology">
  Benchmarks use 1,000-row table operations (create, replace, partial update, select, swap, clear)
  plus a 10,000-row creation stress test. Each framework runs in a fresh Chromium browser context.
  Pyreon and Solid both use template cloning. Vue uses `h()` render functions (not template
  compiler). All frameworks use `createSelector` where available for O(1) selection.
  Run the benchmarks yourself: `npx playwright test e2e/bench-compare.spec.ts`
</Callout>

## What Makes the Difference

### vs React (3-5x faster)

React re-runs component functions and diffs a virtual DOM tree on every state change. Even with
`memo`, it must still check props equality for every row. Pyreon skips all of this — signals
notify DOM bindings directly.

### vs Vue (1.1-1.3x faster)

Vue's proxy-based reactivity knows which components to re-render, but still builds and diffs
VNode trees within each component. Vue's template compiler adds patch flags to skip static
nodes, which narrows the gap. With Vue Vapor mode (upcoming), the gap will likely shrink further.

### vs Solid (~tied)

Solid and Pyreon share the same fundamental architecture: compiled templates, fine-grained
signals, direct DOM updates. The differences are in implementation details:
- Pyreon's `_bind()` tracks deps once (Solid's `createEffect` re-tracks each run)
- Pyreon uses `TextNode.data` for text updates (avoids node destruction)
- Both use `createSelector` for O(1) selection
- Both use LIS-based list reconciliation

These micro-optimizations add up to a consistent ~2-5% edge, but the frameworks are
effectively in the same performance tier.

## Anti-Patterns to Avoid

<Callout type="warn" title="Reading a signal outside a tracking context">
```ts
// Reads once — not reactive
const value = mySignal()
effect(() => console.log(value))  // never re-runs when mySignal changes

// Correct — reads inside the effect
effect(() => console.log(mySignal()))
```
</Callout>

<Callout type="warn" title="Writing signals in a tight loop without batch()">
```ts
// Bad — fires subscribers after each write
for (const item of items) {
  total.update(n => n + item.price)
}

// Good — fires subscribers once after all writes
batch(() => {
  for (const item of items) {
    total.update(n => n + item.price)
  }
})
```
</Callout>

<Callout type="warn" title="Using array.map() for reactive lists">
  Every signal update re-creates all DOM nodes. Use `<For>` with a `key` function instead.
  See the [Lists](/docs/lists) page.
</Callout>
