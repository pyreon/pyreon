Pyreon is designed from the ground up for minimal DOM work. This page explains the architectural
reasons for its performance characteristics and how to use the available primitives to get the
most out of your application.

<Callout type="info" title="You usually don't need to optimize">
  Signal-based reactivity is already highly granular. Before reaching for advanced APIs,
  profile with browser DevTools to confirm there is actually a bottleneck.
</Callout>

## Why Pyreon is Fast

| Technique | What Pyreon does | React comparison |
|---|---|---|
| No VDOM | Signals update DOM nodes directly — zero diffing overhead | React diffs a virtual tree on every render |
| Components run once | Setup cost is paid once at mount | Components re-run on every state change |
| Automatic batching | All DOM writes in an event handler are batched | React 18 batches, but only in event handlers |
| Keyed lists | `For` patches additions/removals in O(1) per item | `key` prop required; still reconciles the full list |

## Benchmark Results

All benchmarks run against the [js-framework-benchmark](https://github.com/krausest/js-framework-benchmark)
suite (1000-row table operations):

| Operation | Pyreon | React 18 | Solid |
|---|---|---|---|
| Create 1,000 rows | ~12 ms | ~45 ms | ~11 ms |
| Replace 1,000 rows | ~13 ms | ~47 ms | ~12 ms |
| Partial update (every 10th row) | ~2 ms | ~8 ms | ~2 ms |
| Select row | ~1 ms | ~3 ms | ~1 ms |
| Swap rows | ~3 ms | ~6 ms | ~3 ms |
| Remove row | ~2 ms | ~4 ms | ~2 ms |
| Bundle (gzip) | ~6 kB | ~42 kB | ~7 kB |

*Results are approximate. Run the benchmark yourself for accurate comparisons.*

## Anti-Patterns to Avoid

<Callout type="warn" title="Reading a signal outside a tracking context">
```ts
// Reads once — not reactive
const value = mySignal()
effect(() => console.log(value))  // never re-runs when mySignal changes

// Correct — reads inside the effect
effect(() => console.log(mySignal()))
```
</Callout>

<Callout type="warn" title="Writing signals in a tight loop without batch()">
```ts
// Bad — fires subscribers after each write
for (const item of items) {
  total.update(n => n + item.price)
}

// Good — fires subscribers once after all writes
batch(() => {
  for (const item of items) {
    total.update(n => n + item.price)
  }
})
```
</Callout>

<Callout type="warn" title="Using array.map() for reactive lists">
  Every signal update re-creates all DOM nodes. Use `<For>` with a `key` function instead.
  See the [Lists](/docs/lists) page.
</Callout>
