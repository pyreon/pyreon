# Code Style Rules

## Linting & Formatting

- **oxlint** for general JS/TS linting (400+ rules, Rust-powered)
- **oxfmt** for formatting (Rust-powered, Prettier-compatible)
- **@pyreon/lint** for Pyreon-specific rules (62 rules, 12 categories)
- Config files: `.oxlintrc.json` (linting), `.oxfmtrc.json` (formatting), `.pyreonlintrc.json` (Pyreon-specific rules)
- Run lint: `bun run lint` (runs `oxlint .` from root)
- Run format: `bun run format` (runs `oxfmt --write .`)
- Check format: `bun run format:check` (runs `oxfmt --check .`)
- Inline suppression: `// oxlint-disable-next-line rule-name` (not `biome-ignore`)
- **oxlint plugin set is CURATED, not wholesale — do NOT enable categories or plugins blindly.** Enabled plugins: `typescript`, `unicorn`, `oxc`, `import`, `jsx-a11y`, `promise`, `node`, `vitest`. Only the **`correctness`** category is on; valuable rules from other categories are cherry-picked individually (`oxc/no-accumulating-spread`, `oxc/no-map-spread` — real O(n²)/spread perf). **Measured fact:** enabling `pedantic` + `jsdoc` wholesale produces **~19,600 findings**, ~99% noise that fights this codebase's deliberate conventions — `no-underscore-dangle` (1,890; Pyreon's `_v`/`_d`/`_s` internals are intentional), `jsdoc/require-param`+`require-returns` (3,400; JSDoc is selective by design), `consistent-function-scoping` (1,065). Even `suspicious`/`perf` wholesale = ~5,300. So: **never enable `pedantic` / `jsdoc` / `suspicious` / `perf` / `restriction` / `nursery` as categories** — cherry-pick the few good rules to `warn`. **Skipped plugins (deliberate):** `react` / `react-perf` (Pyreon components run ONCE — React's re-render-perf rules are semantically wrong here), `nextjs` (not Next), `jest` (repo uses vitest). **Per-rule floods turned off** with rationale: `jsx-a11y/label-has-associated-control` + `control-has-associated-label` (FP-prone; label-association needs cross-element id/scope resolution an AST walker can't do — the documented "high-risk cliff"), `vitest/expect-expect` + `require-mock-type-parameters` + `no-conditional-expect` + `require-to-throw-message` (`expect-expect` mostly FP against the repo's custom assertion helpers). `@pyreon/flow` + `@pyreon/code` are scoped-exempt from the jsx-a11y interactivity rules (`prefer-tag-over-role`, `no-static-element-interactions`, `click-events-have-key-events`, …) — a canvas/diagram + code-editor lib legitimately puts `role`/`onClick` on non-semantic elements. All these fire at `warn` (advisory) — `oxlint .` exits 0 on warnings, so they surface real a11y/perf/async/test signal without gating CI; promote a specific rule to `error` only after driving its findings to zero.
- **oxlint severity is a THREE-STATE model — every rule's state is a deliberate, documented choice; there is no "undecided `warn`".** The governing principle for raising codebase quality through lint:
  - **`error`** = quality is *protected* here. Findings **must be 0**; `oxlint .` exits non-zero, so the `Lint` CI job gates it. This is the only state that prevents regressions.
  - **`warn`** = a *burn-down lane*, not a parking lot. A rule sits here only while its findings are being driven to zero (or because it's an accepted-visible signal with a stated reason). A rule parked at `warn` forever with no intent is the worst state — noise **and** no protection. **The `warn` backlog is ratcheted (see below), so it can only shrink.**
  - **`off`** = deliberately N/A, with a one-line rationale (Pyreon convention, FP-prone, or wrong semantics for this codebase).
  The promotion unit is **atomic fix-and-gate per rule**: one PR triages a rule's findings (fix real bugs, scope-`off` intentional-in-context, fix-via-config when the rule is mis-set for vitest/etc.), drives it to **zero**, then flips it to `error` in the SAME change. Never leave a rule half-fixed at `warn` — that invites refill. **Before promoting**, run `bunx oxlint .` and confirm the rule shows 0 findings (a promoted rule with a surviving finding breaks CI for everyone).
- **The backlog is at ZERO. `oxlint .` reports 0 warnings + 0 errors** (verified; `lint-baseline.json` = `{ total: 0 }`). Every rule is now in a deliberate state: `error` (gated at 0, can't regress), or `off` (verified FP/intentional, with rationale), or scoped-`off` in a legitimate context. The journey was 559 → 0 across the correctness tier + ratchet + burn-down PRs. Do NOT "improve" this by flipping `off` rules back on without re-reading their findings — they were each triaged.
- **Error-gated rules (findings driven to 0):** `oxc/no-accumulating-spread` (real O(n²) — fixed in rocketstyle by mutate-accumulator), `no-unsafe-finally`, `no-unsafe-optional-chaining` (`(x?.y as T)?.z`), `no-control-regex`, `no-unused-expressions`, `no-unused-vars`, `no-useless-escape`, `no-shadow`, `import/no-duplicates`, `unicorn/no-thenable`, `unicorn/prefer-string-starts-ends-with`, `unicorn/no-useless-length-check`, `unicorn/no-useless-fallback-in-spread`, `jsx-a11y/no-redundant-roles`, `vitest/valid-expect` (`maxArgs: 2` — vitest's `expect(actual, message)` is valid; do NOT delete the authors' failure-diagnostic messages), `vitest/no-standalone-expect`, plus `no-console: ["error", { allow: ["warn", "error"] }]` (library diagnostics permitted; stray `console.log` flagged) and `no-debugger`.
- **Rules `off` after triage proved them false-positive/intentional FOR THIS CODEBASE** (verify-first — blanket-fixing any of these would have shipped bugs or churned correct code): `oxc/no-map-spread` (the immutable-update idiom `arr.map(x => ({ ...x }))` — mutating breaks signal reactivity), `unicorn/no-new-array` (deliberate hot-path preallocation — `new Array(CAP)` ring buffers; `Array.from` regresses perf), `no-template-curly-in-string` (doc strings that literally show `${...}` template syntax), `promise/no-callback-in-promise` (legit `.then(ok, err)` Express-middleware bridges + functional store updates), `typescript/no-this-alias` (deliberate `this`-capture in the signal hot path + compat shims), `unicorn/no-useless-spread` (defensive `[...coll]` snapshots — iterating while deleting from the same Map/Set), `typescript/no-non-null-assertion` (`!` is an idiomatic, deliberate TS escape-hatch used ~296× where null is provably impossible after guards; forcing `if (!x) return` everywhere is high-churn high-risk), plus the per-rule floods (`vitest/expect-expect`, `jsx-a11y/label-has-associated-control` + `control-has-associated-label`).
- **Scope-`off` overrides (legitimate context, not FP everywhere):** `no-console` off in `examples/**` + `packages/**/scripts/**` + CLI packages (console IS the output); `jsx-a11y` interactivity/role set off in `examples/**` (illustrative demos) + `packages/ui/primitives/src/**` (headless primitives provide a11y via ARIA-helper props + keyboard handlers the static AST can't see — the real a11y gate is `@pyreon/lint`'s frontend rules + runtime); `vitest/no-disabled-tests` + `no-conditional-tests` + `valid-title` off in `**/*.test.*` (intentional `it.skip` contract-locks + data-driven test registration); `unicorn/no-empty-file` off in `**/*.d.ts` (documented placeholder declaration files); `unicorn/no-thenable` off in the native compiler (`then` = if/ternary IR field). **The recurring lesson (5×): before gating OR forcing a fix, READ the findings — `no-thenable`/`valid-expect`/`no-control-regex`/`no-map-spread`/`no-useless-spread` all *looked* fixable but were correct code the rule mis-flagged.**
- **The `warn` backlog is RATCHETED — `lint-baseline.json` + `scripts/check-lint-ratchet.ts` (CI gate `Lint Ratchet`, also in `validate-fast` + pre-push).** This is the keystone that makes the config self-sustaining instead of a one-time snapshot. Each `warn` rule has a baseline count; a change that pushes any rule **above** its baseline FAILS CI. So new code can't refill the backlog while it's being burned down. The baseline only moves **DOWN**: after fixing findings, run `bun run check-lint-ratchet -- --update` (or `bun scripts/check-lint-ratchet.ts --update`) and review the diff — **never raise a count to absorb new findings** (fix or scope-with-rationale instead). `error`-severity rules are NOT in the baseline (oxlint already gates them at 0). Pure logic (`countWarnFindings` / `compareToBaseline` / `buildBaseline`) is unit-tested in `packages/internals/test-utils/src/tests/lint-ratchet.test.ts`; the gate itself is bisect-verified (lowering a baseline count by 1 → exit 1). Combined with the error tier, **the lint surface can only improve over time** — that's the whole "smart config" mechanism.
- **The SAME three-state + ratchet discipline now covers `@pyreon/lint`'s OWN rules over framework `packages/*/src` — `pyreon-lint-baseline.json` + `scripts/check-pyreon-lint-ratchet.ts`** (CI gate runs in the same `Lint Ratchet` job, also in `validate-fast` + pre-push). The `Pyreon Lint Gate` (`pyreon doctor --only lint --ci`) is the ERROR tier (gated at 0); this ratchet locks the **advisory `warning` + `info`** backlog the framework's dogfooding produces, so it can only shrink. **Key nuance vs oxlint: many of the seed-baseline findings are the expected `off` case, not real debt.** Several `@pyreon/lint` rules are USER-APP best-practices the FRAMEWORK is the legitimate exception to — e.g. `no-raw-addeventlistener` fires on `@pyreon/hooks` (the wrapper layer that's SUPPOSED to call raw `addEventListener`). The fundamentally-correct burn-down for those is to **scope the rule off for that package in `.pyreonlintrc.json` with a rationale** (which lowers the baseline on `--update`), NOT to churn correct framework internals. Rules that genuinely apply to framework code (`prefer-isserver`, `no-error-without-prefix`) stay as burn-down targets → driven to 0 → promote to a gate `error`. Seeded at 344 advisory findings / 22 rules. Pure logic (`countPyreonFindings` / `buildBaseline`, reusing `compareToBaseline`) unit-tested in `pyreon-lint-ratchet.test.ts`; bisect-verified.
- **Pyreon-lint rule options.** Config entries accept a bare severity (`"error"`) or a `[severity, options]` tuple. Rules that support path-based exemption read `options.exemptPaths: string[]` — each entry is a substring match against the file path. The monorepo's `.pyreonlintrc.json` at repo root configures exemptions for DOM-runtime / server-only / cleanup-wrapper-foundation packages (e.g. `packages/core/runtime-dom/` is exempt from `no-window-in-ssr`). **Do NOT hardcode monorepo-specific paths in rule source** — they ship to user apps as dead code at best, leaked internals at worst. Use the `exemptPaths` option instead.

## TypeScript

- `exactOptionalPropertyTypes` enabled — optional properties need explicit `| undefined`
- `jsx: "preserve"` with `jsxImportSource: "@pyreon/core"` in root tsconfig
- `customConditions: ["bun"]` in root tsconfig for workspace resolution
- No build step needed for development — `"bun": "./src/index.ts"` in each package's exports

## Conventions

- Prefer `signal<T>()` callable pattern (not `.value` getter/setter)
- Components are plain functions: `ComponentFn<P> = (props: P) => VNodeChild`
- `onMount` returns `CleanupFn | undefined` (not `void`)
- Use `h()` or JSX — both produce VNodes
- `<For>` uses `by` prop (not `key`) because JSX extracts `key` specially
- Context: `provide(ctx, value)` — pushes context and auto-cleans up on unmount
- `ExtractProps<T>` extracts props from a ComponentFn or passes through
- `HigherOrderComponent<HOP, P>` for typed HOC patterns

## UI Components (@pyreon/ui-components)

- **Layout in `.attrs()`, CSS in `.theme()`**: Element layout props (`direction`, `alignX`, `alignY`, `gap`, `block`, `tag`) go in `.attrs()`. Visual styles (colors, spacing, borders, shadows) go in `.theme()`.
- **Pseudo-state styles**: Use `hover: { ... }`, `focus: { ... }`, `active: { ... }`, `disabled: { ... }` objects inside `.theme()` callbacks. The `el`/`txt` bases handle CSS pseudo-selector generation via `makeItResponsive`.
- **Hover CSS is unconditional**: `:hover` styles apply to ALL components that define hover theme — not just interactive ones (onClick/href). The `cursor: pointer` is the only thing gated on interactivity.
- **CSS property naming**: Use unistyle convention (`borderWidthTop`, `borderColorLeft`) NOT CSS-spec order (`borderTopWidth`, `borderLeftColor`). Property-first naming.
- **useBooleans: false**: Rocketstyle dimensions accept string values (`state="primary"`, `size="large"`), not booleans. This is the default.
- **Size dimensions should be comfortable**: Menu items, dropdown options, and interactive list items need adequate padding. `t.spacing.small` (8px) vertical / `t.spacing.medium` (12px) horizontal is the minimum for touch-friendly sizing.
- **Use `@pyreon/hooks` instead of manual event handling**: Use `useEventListener` for document/window listeners, `useScrollLock` for scroll locking, `useClickOutside` for click-outside detection. Never use raw `addEventListener`/`removeEventListener` in primitives.
- **Use `@pyreon/elements` Overlay for tooltips/popovers/dropdowns**: The `useOverlay` hook handles positioning, viewport flipping, ESC key, click-outside, scroll tracking, hover delay. Never reimplement overlay positioning logic in primitives.
- **Semantic HTML in `.config({ component })`**: Use `component: 'hr'` for Divider, `component: 'nav'` for nav containers etc. This sets the outer rocketstyle element tag. `tag` in `.attrs()` sets the Element's inner tag.
- **Static ARIA landmark/role/name defaults belong in `.attrs()`**: a presentational component that owns no `@pyreon/ui-primitives` behavior base (spinner, tooltip bubble, nav landmark) should carry its WAI-ARIA role/name as a STATIC default in `.attrs()` — `Loader` → `{ role: 'status', 'aria-label': 'Loading' }` (a nameless spinner is invisible to AT; `role="status"` = polite live region + the label names it), `Pagination` → `{ 'aria-label': 'Pagination' }` (a `<nav>` landmark needs a distinguishing name — APG Pagination pattern), `Tooltip` → `{ role: 'tooltip' }` (the trigger still owns `aria-describedby`), `Breadcrumb` → `{ tag: 'nav', 'aria-label': 'Breadcrumb' }` (a List-base whose root Element is a styled flex `<div>` — `tag: 'nav'` converts it to the landmark cleanly; mark the current `<BreadcrumbItem>` with a `aria-current="page"` string). These are DEFAULTS a consumer overrides by passing their own value (direct props win over `.attrs()` via `mergeProps`), and they must be STATIC strings (no `undefined`/boolean branch) so they render correctly through BOTH the runtime `applyProps` path AND the compiled template path. Interactive state ARIA (`aria-checked`/`aria-selected`/`aria-expanded`) is NOT a ui-components concern — it's DELEGATED to the `*Base` primitive. Test with value assertions (`getAttribute('role') === 'status'`), never `hasAttribute`.
- **Severity-driven live-region roles use the `.attrs((props) => …)` CALLBACK form, not a hardcoded static role.** For a message component with a `state`/severity dimension (`Alert` — `info`/`success`/`warning`/`error`), the WAI-ARIA politeness depends on severity: `error`/`warning` should INTERRUPT (`role="alert"` = assertive), `info`/`success` should NOT (`role="status"` = polite). Rocketstyle's `.attrs()` accepts a callback `(props, theme, helpers) => ({…})` (typed via `.attrs<{ state?: … }>(...)` so the dimension prop reads without a cast) that legitimately reads prop VALUES at setup — so bind the role: `assertive = props.state === 'error' || props.state === 'warning'; return { role: assertive ? 'alert' : 'status', 'aria-live': assertive ? 'assertive' : 'polite', … }`. Set `aria-live` explicitly ALONGSIDE `role` (belt-and-braces): an alert/notification container commonly pre-exists in the DOM and its CONTENT mutates while mounted, where explicit `aria-live` announces more reliably than the role's implicit live region (this is the delta from `@pyreon/toast`'s `<Toaster>`, which relies on the implicit live region because its rows are INSERTED, not mutated in place). The callback reads `state` ONCE at setup (like every `.attrs()` callback) — a severity fixed at creation is correct; a runtime severity flip that must move the role needs an explicit `role`/`aria-live` prop. Consumer override is automatic (props win over the callback result via `mergeProps`) — no need to read `props.role` in the callback. An AMBIENT card (`Notification`, toast-like) is the exception: keep it fixed `role="status"` + `aria-live="polite"` regardless of severity — it's a passive surface that should never interrupt by default. Test the severity mapping with value assertions per state (`error`/`warning` → `'alert'`, else `'status'`) plus an override spec.

## UI Primitives (@pyreon/ui-primitives)

- **Use `useControllableState`**: Every primitive with controlled/uncontrolled state must use `useControllableState({ value, defaultValue, onChange })` from `@pyreon/hooks`. Never duplicate the `isControlled + signal + getter` pattern.
- **Shared keyboard navigation**: Use `navigateByRole()` from `keyboard.ts` for arrow key navigation between siblings (tabs, radios). Accepts `containerSelector`, `itemSelector`, and `keys` ('horizontal'|'vertical'|'both'); it also handles `Home`/`End`. For **type-to-select** (WAI-ARIA listbox/tree typeahead), `keyboard.ts` also exports `createTypeahead(resetMs?)` (a per-instance printable-char buffer that resets after ~500ms idle — create ONE per primitive) + the pure `typeaheadMatch(labels, buffer, currentIndex)` (case-insensitive prefix match; a single repeated char cycles from `currentIndex+1`, a longer buffer refines from `currentIndex`). ComboboxBase (`onKeyDown`) and TreeBase (`onKeyDown`) both consume these — do NOT hand-roll a parallel typeahead. For the "no active item yet" case (closed combobox / unfocused tree) pass `currentIndex = -1` so the first keystroke lands on the first match. Any container-level keydown handler that owns shortcuts must bail when `e.target` is an editable field (INPUT/TEXTAREA/SELECT/contentEditable) — the render-function primitives attach `onKeyDown` per item so they don't hit this, but a wrapper that owns tree/list shortcuts does.
- **Render-function primitives provide ARIA helpers**: ComboboxBase exposes `inputProps()`, `listboxProps()`, `getOptionProps()`. TreeBase exposes `treeProps()`, `getItemProps()`. FileUploadBase exposes `inputProps`. CalendarBase exposes `gridProps()`, `rowProps`, `columnHeaderProps`, `getDayProps(day)` (WAI-ARIA date-grid: `role="grid"`/`row`/`columnheader`/`gridcell`, full-date `aria-label`, `aria-selected`/`aria-current`, one roving `tabIndex=0`). Always add ARIA helper objects to state for render-function primitives. **ARIA STATE values are STRINGS, not booleans** (`'aria-selected': day.isSelected ? 'true' : 'false'`) — a boolean `aria-selected={true}` is otherwise presence-only `aria-selected=""`, which assistive tech does NOT read as "true". The runtime `applyStaticProp` (`packages/core/runtime-dom/src/props.ts`) now coerces boolean `aria-*` → `"true"`/`"false"` as a SAFETY NET (aria-first, BEFORE the generic boolean-presence branch; SSR mirrors it). **The recommended `aria-x={cond ? 'true' : undefined}` shape now works identically through the compiled JSX template fast path too** — the compiler's `attrSetter` routes generic dynamic attrs through the runtime `_setAttr` (= `applyAttrProp`), which mirrors `applyStaticProp` (null/undefined → removeAttribute, boolean-aria → "true"/"false", boolean → presence/absence). Before that fix the template path emitted a raw `setAttribute`, so the `undefined` branch rendered `aria-x="undefined"` in real (vite-plugin-compiled) apps while the `h()`/browser-test path was fine — see anti-patterns "Compiler template fast-path value emit diverging…". Return the literal `'true'`/`'false'` string, or `undefined` to omit. (As of the 2026-07 boolean-aria audit, EVERY `@pyreon/ui-primitives` aria-state emission is a string — the follow-up audit is CLOSED; the last residual booleans were the `aria-invalid` sites (Switch/Select/Checkbox/Slider/RadioGroup), Slider's `aria-disabled`, and Tree's `aria-multiselectable`, all written as `x || undefined`. New helpers must continue to use strings.)
- **WAI-ARIA keyboard nav is complete for the interactive primitives** (the #2212 boolean-aria follow-up that left "ComboboxBase/TreeBase lack Home/End + typeahead; Tree lacks `*`" is now **CLOSED**): ComboboxBase's `onKeyDown` handles ArrowUp/Down + `Home`/`End` + Enter/Escape/Tab + **typeahead** (listbox pattern); TreeBase's `onKeyDown` handles ArrowUp/Down/Left/Right + Enter/Space + `Home`/`End` + **typeahead** + **`*`** (expand all siblings at the focused node's level — tree pattern). Radio/Tab get Home/End via `navigateByRole`; Slider/Modal keyboard were already complete. Residual (tracked, NOT in this pass): TreeBase `ArrowLeft` on a collapsed/leaf node does not yet move focus to the parent (WAI-ARIA tree "move to parent" — a separate enhancement, not a correctness bug).
- **Reactive conditional rendering**: Use `return (() => { if (!cond()) return null; return <div>...</div> })` not `if (!cond()) return null; return <div>...</div>`. Components run once.
- **No `as unknown as VNodeChild`**: JSX.Element (VNode) is assignable to VNodeChild. The cast is unnecessary.

## Dead Code

- Remove dead code rather than commenting it out
- If browser APIs are removed from spec, remove the code that uses them
- Don't add backwards-compatibility shims for removed features
- **An "unused import/variable" finding is a SYMPTOM, not a verdict.** It usually means someone intended a behavior and never finished wiring it — ask *why the symbol was written* before deleting it. Of four such code-quality-bot findings on the 2026-07 audit PRs, following the advice literally would have made three worse: (1) `const { result, unmount } = mountWith(...)` with `unmount` never called was a **leak** — `mountWith` has no auto-cleanup and every sibling call site unmounts, so the mounted component, its subscribed effect, and the container element leaked into later tests; the fix was to CALL it. (2) `VERCEL_ADAPTER_OUTPUT` imported but referenced only inside a comment (in two PRs) was a **missing assertion** — that file exists to drift-lock scaffolded deploy configs against the adapters' real output constants, and vercel was the one adapter left unasserted; deleting the import would have cemented the hole. Only the third (`afterEach`, zero call sites) was genuinely dead. In tests especially: an unused cleanup handle means a leak; an unused constant in a contract test means an unasserted contract. Record the reasoning in a comment so the next analyzer pass doesn't re-suggest the deletion.
