# Codebase audit — 2026-04-27

## Why this exists

Recent sessions have been adding code (perf optimizations, infra, RFCs) on top of an already-large surface area that **keeps producing real-app regressions**. Before more additions, this audit asks: what do we actually have, who uses it, what keeps breaking, where should we invest?

**Framing note (revised after the first audit pass).** The first cut of this doc proposed archiving / removing low-consumer packages. That framing was wrong. Every package in the workspace was built intentionally — they're products, not bloat. "0 internal consumers" for a published library means "we don't have a demo example", not "this is dead code". The audit data is still useful; the recommendations have been rewritten to focus on **investing better in what we have**, not removing it.

### Tier C — Published libraries lacking demo apps (validation gap, NOT removal candidate)

**Initial framing of this section was wrong.** The first audit pass labelled these as "marginal" because no example app imports them. That conflates "nobody uses this internally" with "this isn't valuable" — wrong for **published** libraries whose target audience is **external users** (apps consuming `@pyreon/*` from npm).

Every package in the workspace was built intentionally. Low internal consumer count for a published library is a **demo / documentation gap**, not a removal signal. The right action is to add an example that exercises the surface, so:
- Future framework changes can't silently break it
- Users adopting it have a working reference
- The package shows up in the verify-modes matrix

## Recommendations — concrete, ordered by ROI

**Reframed after the first audit pass.** None of these are about removing code. Every package was built intentionally. The recommendations are about **investing better** in what we have: better demos, better bundle hygiene, better positioning, better stability focus.

### High ROI (do first)

1. **Investigate the `@pyreon/document` 14MB lib.** The lazy-load story is documented (`Heavy renderers lazy-loaded — PDF ~300KB, DOCX ~100KB, XLSX ~500KB, PPTX ~200KB`) but the lib total is 14MB. Either the lazy-loading isn't working in production builds or the doc is out of date. Either is a real bug worth fixing — affects every app that uses `@pyreon/document`.
2. **Investigate the `@pyreon/charts` 9MB lib.** Same question — ECharts tree-shaking story documented, 9MB landed. Real bug if lazy-loading is broken.
3. **Update `CLAUDE.md`'s "fastest framework on all benchmarks" claim.** Honest version: "competitive with Solid on synthetic benchmarks, no real-app head-to-head yet." The current claim is overstated and erodes trust if a reviewer measures.
4. **Add minimal example apps for the 9 demo-gap packages** (Tier C above). Each is a ~50 LOC `examples/<pkg>-demo/` that imports the surface, mounts something, and gets exercised in the `verify-modes` matrix. Cheap insurance against silent rot. Order by published-package value: dnd, feature, react-compat, vue-compat, solid-compat, preact-compat, connector-document, rx (more coverage), storybook (if internal demo makes sense).

### Medium ROI (do next)

5. **Stabilize the rendering / styling layer.** 5 packages (runtime-dom, styler, rocketstyle, elements, unistyle) produce 24% of fix-commits on 13% of the LOC. They're load-bearing AND fragile. Concrete: add a real-app regression suite that runs against bokisch.com or a clone before PRs to these packages merge. Synthetic tests have proven insufficient (PRs #197, #200, #336, #349 all surfaced regressions only real apps caught).
6. **Audit `@pyreon/lint` rule fire-rate.** 59 rules, 6,915 LOC. Find which rules actually fire in real apps (the showcases). Rules that haven't fired in 6 months either aren't catching anything or aren't being violated — both signal scope reduction is worth considering. **This isn't archive — it's "shrink the maintained surface to the rules that earn their keep".**
7. **Audit `@pyreon/hooks` hook-by-hook.** 35 hooks. Find which hooks have 0 consumer references in examples + showcases. Same question as for lint rules: a hook that's never used isn't wrong, but it's not validated either. Fix is "use it" (build a demo) not "delete it".

### Low ROI (consider later)

8. **Set bundle-size budgets per package.** Once measurements exist (we have `libKB` now), define ceilings. PRs that break the budget need justification. Catches regressions in lazy-load, dynamic-import boundaries, dep additions.
9. **Set fix-rate thresholds.** A package with 20 fixes in 6 months is signaling something — either it's load-bearing AND stable design (rare), or it's under-designed. Triage the top-5 fix-rate packages quarterly.
10. **Add a "real app smoke" gate.** PRs to Tier A packages should pass a real-app smoke (bokisch.com or a clone). Same point as #5, scoped as a CI gate.

## Aggregate insight

Of 100,484 source LOC across 61 packages:
- **~13,000 LOC (13%) in 5 high-churn foundation packages** (Tier A: runtime-dom, styler, rocketstyle, elements, unistyle) — where 24% of recent fix commits land. **Stability investment here has the highest leverage.**
- **~9,000 LOC (9%) in published libraries that lack internal example apps** (Tier C). Not slack — these are intentional product surface. The risk is silent rot when framework internals change. **Investment: minimal demo apps that the verify-modes matrix exercises.**
- **23 MB of compiled lib in 2 packages** (document 14MB, charts 9MB) where lazy-load / tree-shake is documented but apparently not landing. **Investigation: real bundle bug.**
- The "fastest framework on all benchmarks" claim in `CLAUDE.md` is **not currently backed by data.** Tied with Solid; no real-app head-to-head ran.

**The audit's main finding, corrected:** the framework's surface is intentional — every package has a place. The leverage isn't in removing code, it's in (a) hardening the rendering/styling foundation where real-app regressions concentrate, (b) closing the demo-app gap so published libraries don't silently rot, (c) fixing the bundle-size bugs in document and charts, and (d) updating positioning to match what we can actually measure.

## What this audit does NOT cover

- **Bundle size after tree-shake at consumer level.** We measure `lib/` but apps tree-shake. A 14MB lib might be 1MB in a real app build. Worth measuring properly.
- **Real-app regression coverage.** The 371 fix-rate counts commits that LANDED. It doesn't count regressions that the test suite missed and that a real app caught later.
- **Documentation completeness.** Some packages have `manifest.ts` (driving generated docs), some don't.
- **Test quality vs quantity.** A package with high test/src ratio could be over-mocked or under-real-shape. The mock-vnode audit (PR #311 area) hits this.

These are follow-up audits if needed.
